MCP是 Model Context Protocol 的缩写, 本质 是一个协议标准。 定义了 Agent 或 LLM 如何与外 部工具 数据源进行交互。 他统一了工具调用的格 式 鉴权 生命周期管理。 它作用把各种不同的工 具 (本地文件 数据库 API 浏览器等) 抽象成 一套标准化的接口, 这样 Agent 不需要为每个工 具写定制的调用逻辑。

Skill我个入理解它是更高层的封装。 Skill 往 往包含多个步骤 多个工具调用的组合; 甚虿包含 一些 prompt 模板和业务逻辑。 比如 66 发送邮 件"是一个工具 (通过 MCP 暴露) , 但 撰写一 封得体的会议纪要邮件并发送"' 就是一个 Skill, 它可能包含: 调用 LLM 生成内容 >调用邮件工 具发送 >调用日历工具更新状态。 Skill 是面向任 务的。可复用的能力单元。

Prompt是直接输入给 LLM 的文本指令。 但它和 前面的概念会交叉: 一个 Skill 内部会包含多个 Prompt。 比如规划用的 Prompt 工具选择用的 Prompt 结果总结用的 Prompt。 Skill 本质上就 是 "Prompt + 工具调用编排 +控制逻辑" 的组 合。

CLI 是指 Command Line Interface。 在 Agent 系统中 CLI 可以有两个角色: 一是 Agent 可以 去调用的命令行工具;, 这时它算是一种特殊的 T001; 二是 Agent 系统本身的交互入口; 比如通 过命令行启动一个 Agent 会话。 我更倾向从 'Tool 的角度去理解它-_CLI 是暴露给 Agent 的一种

可执行能力, 通过 MCP 包装后, Agent 就能理解 和调用它。

Harness, 我理解它是Agent 的框架环境。 比如 你写好了一个 Agent = 9 它包含各种 Skills Prompt Tool 配置; 但你需要在一种可控的环境 里去跑它 评估它的表现调试它的行为。在 Agent场景里 Harness可以批量运行Agent并收 集执行轨迹, 用内置或自定义的指标做自动化打 分。也可以接入LLM-as-Judge做更灵活的定性评 估。 它能帮助开发者系统化地做回归测试和4/对 比; 在迭代Agent时快速判断新版本有没有退化。

它们的关系是: Harness 评测 Agent。 Agent 调 用 Skill, Skill 通过 MCP 调用 CLI 等底层工具; Prompt 贯穿全程。

文档 一〉切 chunk 一> embedding >向量库 top-k 检索 >拼进 prompt 一> LLM 生成答案

这个流程适合 FAQ 产品文档问答- 知识库检索这 类场景。

它的价值是把模型从 只靠参数记忆回答" ,娈 成 "能引用外部资料回答"

面试官: 那传统 RAG 最大的问题是什么?

答:

传统 RAG 的问题不是完全不能用 , 而是它经常 把 '文档工作" 简化成 文本片段召回"

备忘录

短问答里这没问题。

但一旦进入复杂文档场景,就会出现三个典型问题 0

第一, 结构容易丢。

一份合同里的某一条, 不只是几句话。 它属于哪个章 节。引用了哪个定义:下面有没有例外`附件里有没 有补充说明, 都会影响含义。

如果直接切成差不多长度的 chunk, 模型可能看到了 文字,却看不到它在文档里的位置。

第二,关系容易丢。

正文可能引用表格 ,表格里有金额和日期; 主合同可 能被补充协议修改; 扫描件里可能有签章页。

flat chunk 很难保住这些关系。

第三, 很难复核。

很多业务场景里 , 答案好看不够c 0 系统必须能回到原 文,说明这个结论来自哪份文件: 哪一页。哪一节; 最好还能继续展开上下文。

所以传统 RAG 的核心短板可以概括成-句:

备忘录

它能找回文本,但不一定能保住文档状态。

面试官: 什么叫文档状态?

答:

文档状态指的不是单纯的文本内容 而是文档在系统 里的结构。位置 关系和证据路径。

比如分析一批合同,判断哪些条款可能因为交易触发 consent 或 notice。

普通 RAG 可能只返回几段相似文本。

但真正可用的系统,至少应该知道:

这段文字来自哪份合同

它属于哪个章节

附件里有没有相关表格

补充协议有没有改过这条

这个结论能不能回到原文复核

所以 document state 更像这样:

备忘录

Document Section Chunk Asset Metadata > Source Path 一〉 Graph Link Retrieval Trace

Agent 拿到的就不是一堆孤立纸条,而是一个可以继 续探索的文档空间 0

面试官: DeepResearch 为什么看起来 不像传统 RAG?

答

DeepResearch 更接近 Agentic RAG。

传统 RAG 是先把固定知识库切好 chunk, 每次问题 来了就做 top-k 召回

DeepResearch 的流程更像研究员做任务:

提出问题 〉 搜索网页 >打开来源 一〉 筛选信息 一〉 继续追问 一>汇总证据 >生成报告

它不是完全不需要检索。

区别在于,检索不再只是外部模块 , 而变成 Agent 推理过程中的一个动作。

备忘录

模型可以决定下一步搜什么`打开哪个页面。保留哪 条证据 什么时候继续查。

所以 DeepResearch 代表的方向不是 RAG 消失 而是:

RAG 从 '静态召回 走向 '动态研究"

面试官: 那 Agent 会取代 RAG 吗?

答

不会简单取代,更可能是融合。

未来复杂问答系统里, 大概率会同时存在三层能力:

Long Context: 让模型一次读更多材料

Retrieval: 让系统能从外部材料里找证据

Agent: 让模型能决定下一步查什么`怎么查 查完怎么继续

真正的难点不是把这三个词放进架构图 0

难点是让 Agent 操作的对象更可靠。

备忘录

如果 Agent 每一步拿到的都是几个弧立 chunk, 它 就很难像人一样在文档里移动。它只能不断重新检 索。重新拼上下文。重新猜关系。

所以 Agentic RAG 的关键是:

不只是让模型会调用搜索工具。而是给它-个可 以导航。可以引用 可以更新的文档状态。

面试官: 如果让你设计复杂文档 RAG, 你 会怎么做?

答

不会直接从 chunk size 和 embedding model 开 始调。 0

复杂文档 RAG 最容易出问题的地方 , 往往在更前 面: 文档进入系统时,结构 来源。表格。图片。附 件关系有没有被保留下来。

所以会先加一层 document memory。

这层不负责直接生成最终答案,而是负责把文档整理

备忘录

成 Agent 可以工作的状态:

文档要有层级,不只是平铺 chunk

chunk 要带路径` 来源和上下文

表格 图片 签章页 附件不能和正文断开

检索结果要能回到原文和章节

Agent 要能沿着文档结构继续展开, 而不是每 次重新 top-k

这个思路可以参考一个开源项目 Ontos- ailknowhere。
