## 一面

Q1：自我介绍。

A：硕士毕业一年，简单讲了下过往经历和技术栈，主要是在 AI Agent 这一块。

Q2：面试官挑了一个简历里自动标注视频的项目，问技术架构、数据链路、以及成果。

A：大概分三步：

1. 降噪转录（把冗余语气词干掉并做分钟级总结）；
    
2. 词表聚类和路由（几万个词聚类成十几个大类，分发到不同工作流）；
    
3. 打标过滤（处理词表太大带来的大模型 Attention 丢失问题，用 Self-consistency 打乱词表跑十遍取并集，最后用打分模型过滤）。
    

Q3：（追问）降噪的具体处理，是先切分分钟级再去总结吗？

A：不是，是先用大模型拿到完整的逐字逐句 Transcript，再去按分钟总结。因为直接切分总结会丢失代词前后的上下文信息。

Q4：（追问）拿原始 Transcript 用的是什么模型？

A：试过 Whisper 但是发现精度不行，最后用的 Gemini Pro，因为它的 Context window 足够大。

Q5：（追问）长达几小时的视频是怎么喂给大模型的？

A：用 Sliding Window（滑动窗口），以固定时长为节点去读视频，然后拼接起来。

Q6：（追问）喂入后，输出格式是什么样的？

A：输出的是全量文本，没有用 output schema，时间戳是用其他工程方法打上去的。

Q7：（追问）分钟级文字总结的方向是什么？要输出标签吗？

A：不输出标签，只输出没有噪音的叙述片段。但会保留核心实体，方便后续路由到不同的标签分类器。

Q8：（追问）词表命中后路由到哪个 workflow，同一分钟会有多个 workflow 去处理吗？

A：会有多个 workflow 同时处理。

Q9：（追问）命中词表的策略是什么？大模型还是小模型？

A：因为分类比较明确，直接用小参数量的大模型做的分类。

Q10：（追问）多个 workflow 的结果在最后怎么 merge？

A：最后在 Filter 之前会做一个单纯的合并。

Q11：（追问）整个这一套是用一个 Leader Agent 串联还是传统服务架构？

A：是一套单纯的 Workflow。因为节点判定明确，不涉及 Planning 的部分。

> [!tip] ClawCodex 项目关联 — Leader Agent vs Workflow
> ClawCodex 的 Orchestrator（`extensions/orchestrator/orchestrator.py`）就是 Workflow 模式：polling → issue 分发 → AgentRunner 执行 → git sync → PR。不是 Leader Agent 编排，因为每个 issue 的处理流程是确定性的。和面试者回答的「节点判定明确，不涉及 Planning」完全一致。
> 
> 但 ClawCodex 更进一步 — 外层 Orchestrator 是 Workflow（确定性 polling + dispatch），内层 AgentRunner 是 ReAct（LLM 自主决定 tool call）。两层的原因：Workflow 控制成本（max_turns / timeout），ReAct 给 agent 灵活度。

Q12：问了另一个处理医疗保单的项目，主要是问当时做的提示词缓存，和普通的前缀缓存有什么区别。

A：主要是通过固定 Prompt 头部内容，Mask 掉用户个人信息或敏感变量，用 SimHash 在数据库里匹配，命中就复用生成的 SQL 或代码，节约推理成本。

> [!tip] ClawCodex 项目关联 — 前缀稳定性缓存（Prefix-Stable Caching）
> `assistant_text.py` 里的流式渲染缓存用的是完全相同的模式：
> 
> ```python
> # 找到流式文本中最后一个"安全断点"前缀
> prefix = _last_checkpoint_prefix(self.streaming_text)
> # 用这个前缀的 SHA-256 作为缓存 key
> cache_key = _content_hash(prefix) if prefix else None
> # 不同的流式尾巴共享同一个缓存 entry
> rendered = cache.get_or_render(self.streaming_text, cache_key=cache_key)
> ```
> 
> 面试者的做法：固定 Prompt 头部 → Mask 变量 → SimHash → 命中复用生成结果
> 我们的做法：固定 checkpoint 前缀 → 忽略流式尾巴 → SHA-256 → 命中复用渲染结果
> 
> 两者本质都是 **前缀稳定性缓存** — 识别出输入中「已稳定的前缀」和「还在变化的尾巴」，只对前缀做 hash 匹配。区别只是一个缓存 LLM 推理结果，一个缓存 Markdown 渲染结果。
> 
> 面试话术：*"这个思路我在 TUI 流式渲染里也用了。流式文本每来一个 chunk，我只对'最后一个安全 Markdown 断点之前的前缀'算 hash 做缓存 key，chunk 间复用渲染结果避免重复解析。本质上和 Prompt 缓存是同一个 prefix-stable caching pattern。"*

Q13：复杂业务审计项目里的"智能分层架构"是怎么做的？有哪些 Agent 逻辑？

A：第一层过 Rule-based filter；第二层用 Coding Agent 读 Data Schema 自己写 SQL 去不同的表抓取特征；第三层对复杂 case 用 RAG 读长文本行业规则；第四层 Agent 自己草拟邮件给人类审核员。

Q14：（追问）这里的"分层"体现在哪里？

A：体现在漏斗状筛选。把计算最昂贵（最耗 Token）的写代码和 RAG 任务放在最后，用便宜的 Rule-based 先挡掉大部分常规请求。

> [!tip] ClawCodex 项目关联 — 漏斗架构
> Orchestrator 的 `_poll_and_dispatch` 就是同构的漏斗：
> 
> ```
> 1. 控制命令检查（零成本）
> 2. Clarification 轮询（零成本）  
> 3. Retry 队列处理（零成本，只是重调度）
> 4. Escalation 处理（零成本）
> 5. Review feedback（轻量 API 调用）
> 6. fetch_candidate_issues（tracker API 调用）
> 7. 逐个 issue 检查：
>    - 已在运行？跳过
>    - 已完成？跳过
>    - 已被 claim？跳过
>    - Intent 检查 + role check（轻量）
>    - 依赖满足？（registry 查询）
>    - ⬇ 到这里才 create workspace + launch agent（最贵）
> ```
> 
> 7 层检查之后才到最贵的 agent 执行。和面试者的漏斗完全同构，只是一个过滤审计 case，一个过滤 issue。

Q15：（追问）这里的缓存结果存在哪里？

A：存在云端关系型数据库里。

Q16：换了一个算法项目问，大模型 Post-training（LoRA 微调）做了什么工作？

A：为了满足合规需求，微调模型在识别到用户输入内部密钥等敏感信息时，直接输出固定的拒绝回答，并触发后台抹除数据。

Q17：（追问）LoRA 微调的数据量级是多大？样本分布怎么考虑的？

A：算上合成数据大几千条。真实泄露数据很少，我们用大模型合成了两三千条敏感信息问答数据。

Q18：（追问）有去了解过 PPO 和 DPO 是怎么做后训练的吗？

A：PPO 是强化学习通过反馈尝试小步调整策略；DPO 是从偏好数据直接学奖励模型。在拒绝敏感信息这种场景，用 DPO 收敛更快。

Q19：抛开项目，现阶段一个业务可落地的智能 Agent 应该有哪些模块和范式？

A：想了一下，我觉得最重要的是自学习和自进化能力，因此需要好的长短期记忆模块。此外代码执行能力也很重要，还有联网搜索，其实感觉信息采集这块大多数 agent 做的都不够好。

> [!tip] ClawCodex 项目关联 — 可落地 Agent 的模块清单
> 直接对着 ClawCodex 的架构讲：
> - **Memory** — `src/memdir/`，四种类型（user/feedback/project/reference），文件系统持久化
> - **Tool Use** — MCP 服务器 + 内置 tools（Read/Write/Edit/Bash/Grep/Glob）
> - **Code Execution** — Bash tool + sandbox 沙盒隔离
> - **Permission System** — 四级信任梯度（default/acceptEdits/dontAsk/bypassPermissions）
> - **Orchestration** — 多 issue 并发调度 + retry/followup/review gate

Q20：（追问）为什么现在部分开源 Skills 协议的热度高于 MCP？

A：MCP 格式较重且一次性注入容易导致 Token 爆炸；Skills 是渐进式注入的，Agent 按需读取 Markdown 定义的动作，隔离了无关上下文。

> [!tip] ClawCodex 项目关联 — MCP vs Skills
> ClawCodex 两个都实现了：
> - **MCP** 在 `src/services/mcp/` 里，通过 `ToolSearch` 按需加载 tool schema，不是一次性全注入
> - **Skills** 在 `src/skills/` 里，用户输入 `/skill-name` 时才读取对应 markdown 注入上下文
> 
> 可以用源码说明两者的加载时机差异，完美印证面试者说的「渐进式注入，避免 Context 爆炸」。

Q21：场景设计题：用户提供视频图片素材，怎么搭建一个"智能成片" Agent？

A：先要有一个 Planning Agent，根据用户 Prompt 提取有价值片段；然后交给不同的 Sub-agent 去生成文案；最后指令交给基座模型剪辑。  
（面试官反馈：工业界为了 Latency，不会全靠生成，会大量复用已有的素材库）。

> [!tip] ClawCodex 项目关联 — 语义边界切分（同构问题）
> `assistant_text.py` 的 `_at_safe_checkpoint()` 解决的是同一个问题 — 在流式文本里找语义安全边界：
> 
> ```python
> def _at_safe_checkpoint(text: str) -> bool:
>     # 1. 段落结束 (\n\n)
>     # 2. 围栏代码块关闭 (```)
>     # 3. 标题行结束
>     # 4. 列表项结束
>     # 5. 引用块结束
>     # 6. 水平线
> ```
> 
> RAG chunking 和流式渲染 chunking 面对同一个基础问题 — **在不可预知长度的文本流里，怎么在不丢语义的前提下找到最优切分点**。如果每个 chunk 都触发 re-render，Rich Markdown 解析器会在未闭合的 `**bold**` 或 ` ``` ` 上产生渲染伪影。所以只在语义完整的边界点渲染，和 RAG 里「按 Section 切」是同一个思路。

Q7：简历里的"常识记忆系统"是什么？

A：基于回调机制做的。对话开始前 Retrieve 记忆；对话结束后用轻量模型判断新知识并入库向量库。每晚跑 Cron Job 解决新旧记忆的 Update/Delete/Keep 冲突。

> [!tip] ClawCodex 项目关联 — Memory 系统
> ClawCodex 的 `src/memdir/` 就是这个。基于文件系统的 memory（MEMORY.md 索引 + 独立 .md 文件），对话开始前加载相关记忆，结束后判断新知识写入。和面试者说的「回调机制 + 异步入库」思路相同，但 ClawCodex 用的是文件而非向量库。

> [!tip] ClawCodex 项目关联 — 结构化知识提取
> ClawCodex memory 的写入流程本质上就是这个：对话结束 → 判断哪些信息值得保存 → 提取结构化知识 → 写入 `.md` 文件。
> 
> 但比「问答对」更丰富 — 每条记忆有结构化格式：
> ```markdown
> ---
> name: feedback-no-trailing-summary
> description: user dislikes trailing summaries after code changes
> metadata:
>   type: feedback
> ---
> Rule: 不要在每次操作后写总结
> **Why:** 用户说 "我能看 diff"
> **How to apply:** 改完代码只说改了什么，不要复述
> ```
> 
> 每条记忆有 **name**（去重 key）、**description**（用于判断相关性）、**type**（决定加载时机）、**Why/How**（让 agent 判断边界情况）。

> [!tip] ClawCodex 项目关联 — 自动化 vs 人工的信任梯度
> ClawCodex 的 F-44 review gate 就是这个矛盾的代码实现：
> 
> ```python
> # schema.py
> review_required: bool = False   # True = agent做完必须人审
> auto_approve: bool = False      # True = 跳过审核直接完成
> ```
> 
> Orchestrator 里对应三条路径：
> - `review_required=False` → agent 跑完直接 `mark_completed`
> - `review_required=True, auto_approve=False` → 跑完进 `PENDING_REVIEW`，人跑 `orchestrator issue review --approve` 才放行
> - `review_required=True, auto_approve=True` → 跑完自动放行（给信任度高的场景）
> 
> 再加上 `ApprovalPolicy` 里的 `reject.sandbox_approval` / `reject.rules` 做 tool-level 拦截，整套就是一个可配置的「信任梯度」。
> 
> 面试话术：*"我不是理论上觉得不敢放权，而是在代码里实现了四级信任梯度，从逐个 tool call 审批到完全 bypass。"*

> [!tip] ClawCodex 项目关联 — Coding Agent 底层实现
> ClawCodex 的 `QueryRunner`（`extensions/api/query.py`）跑 ReAct 循环：LLM 输出 → tool call → tool result → 继续。
> 
> 代码检索用 CodeGraph（tree-sitter AST 索引），不是纯 embedding。多 Agent 架构体现在 Orchestrator 把 planning（issue 分发）和 execution（AgentRunner 跑 tool call）的上下文完全隔离。

Q16：如果实现一个多 Agent 协作的写作系统，协作怎么实现？

A：核心是上下文传递（State passing）。传统工作流传递太僵硬；更理想的是依托有基座模型的流式输出，实现 zero-latency 的互相纠偏。

> [!tip] ClawCodex 项目关联 — 多 Agent 上下文隔离与共享
> Orchestrator 的 `max_concurrent_agents` + semaphore 并发控制，每个 agent 有独立 workspace（isolated strategy），通过 `IssueRegistry` 共享状态。
> 
> **真实 bug 案例（F-40）**：多个并发 issue 共享同一个 `ProgressReporter` 实例，导致 issue A 的 `_phase_count` 被 issue B 覆盖，dashboard 显示错误进度。
> 
> 修法是 `_build_session_sink()` 每次为每个 session 创建独立的 `CompositeProgressSink`：
> ```python
> def _build_session_sink(self, task_id: str):
>     inner = ToolContextProgressSink(
>         task_id=task_id,           # 每个session绑定自己的task_id
>         context=self._progress_context,  # 共享底层ToolContext
>         workflow_phases=...,
>     )
>     return CompositeProgressSink([inner])
> ```
> 
> 共享 `ToolContext`（写进度到同一个 store），但隔离 `phase_count` / `current_task_id` 等可变状态。**不是全共享也不是全隔离，而是按字段粒度决定哪些共享、哪些隔离**。

> [!tip] ClawCodex 项目关联 — 什么时候该建图谱
> CodeGraph 走的是相反的路 — **确实建了完整图谱**，但不是用 LLM，而是用 tree-sitter AST 解析：
> - 每个符号（函数/类/变量）是节点
> - caller→callee、import、type-reference 是边
> - 存在 SQLite 里，查询亚毫秒级
> 
> 面试者放弃建图谱是因为数据库表之间的关系是隐式的（外键、业务逻辑关联）。但代码里的关系是**显式的**（函数调用、import 语句），所以 AST 解析能以极低成本建出完整调用图。
> 
> 深层洞察：**什么时候该建图谱取决于关系的显式程度** — 业务数据的关系隐式且多变，建图谱 ROI 低；代码的关系显式且确定，建图谱 ROI 极高。`codegraph_trace` 一次调用就能返回从函数 A 到函数 B 的完整调用链，grep 做不到这件事。

Q5：（追问）这个项目一定要用 Workflow 吗？用 ReAct 可以吗？

A：可以，但 ReAct 会让上下文无限变长。Workflow 能在前期有效切断不必要的上下文，节省算力。

> [!tip] ClawCodex 项目关联 — Workflow vs ReAct 的双层架构
> ClawCodex 两层都有：外层 Orchestrator 是 Workflow（确定性 polling + dispatch），内层 AgentRunner 是 ReAct（LLM 自主决定 tool call）。
> 
> 为什么要分两层：Workflow 控制成本（max_turns / timeout），ReAct 给 agent 灵活度。Orchestrator 的 `_poll_and_dispatch` 是确定性的 7 层漏斗过滤，过滤完才交给 ReAct agent 执行。

> [!tip] ClawCodex 项目关联 — Workflow → Agent 的演化路径
> ClawCodex 的 Orchestrator 正在经历这个演化，能讲出具体的演化路径：
> 
> **纯 Workflow 阶段（最初）**：polling → dispatch → execute → sync，完全确定性
> 
> **混合阶段（现在）**：
> - Intent resolution — agent 的行为由 label + comment command 动态决定（retry / followup / blocked）
> - Clarification — agent 主动向人类提问，根据回答调整行为
> - Review feedback loop — agent 读 PR review comment，自动 follow-up commit
> 
> **往 Agent 演化的阻力**：
> - 确定性丢失 — Workflow 里每个 issue 的处理流程一样，容易 debug；Agent 化后每个 issue 可能走不同路径
> - 成本失控 — Agent 自主决策意味着不可预知的 token 消耗，所以必须有 `max_turns` / `run_timeout_ms` / `max_tools_per_turn` 这些硬约束
> - 状态一致性 — Workflow 的状态机（QUEUED → RUNNING → COMPLETED/FAILED）简单清晰；Agent 化后要处理暂停恢复、中途接管（takeover）、部分完成等中间态
> 
> 面试话术：*"我正在做的项目就处在 Workflow → Agent 的中间态。外层保持 Workflow 的确定性调度保证可观测性，内层用 ReAct 给 agent 灵活度，但必须加 max_turns / timeout / stagnation guard 三重硬约束防止内层 agent 失控。核心张力是：agent 越自由、越需要 harness；harness 越重、agent 越受限。"*

Q11：（追问）看过底层原理吗？

> [!tip] ClawCodex 项目关联 — File System Memory 的隐藏代价
> 面试者讲的是 Claude Code 的 memory 设计，但我们项目里的 `memdir/` 实现揭示了这个设计的深层挑战：
> 
> - **类型化** — 不是一坨日志，而是分 user / feedback / project / reference 四类，每类有不同的写入时机和使用场景
> - **衰减问题** — project 类型的记忆衰减最快（"merge freeze 到周四"，过了周四就是垃圾），所以每条记忆要写 `**Why:**` 和 `**How to apply:**`，让未来的 agent 判断这条记忆是否还有效
> - **验证问题** — 记忆说"函数 X 在 file.py 第 100 行"，但代码可能已经重构了，所以有 "verify before recommending" 规则
> - **去重** — `MEMORY.md` 索引文件限 200 行，超了就截断，所以必须语义去重
> 
> 面试话术：*"File System Memory 的核心挑战不是存取，而是衰减管理和一致性验证。我在项目里给记忆做了四种类型分类，每种有不同的 TTL 特征；读记忆前必须验证它引用的代码实体是否还存在，否则会给 agent 错误的上下文。"*

> [!tip] ClawCodex 项目关联 — AI Coding 的 Edge Case Bug

Q18：有了解过 Harness Engineering（手脚架工程）吗？如何评估后续 Agent 迭代方向？

A：逻辑环境上，Agent 必须有强制反问机制确认任务边界；工程环境上，要隔离多余工具给 Agent 纯净沙盒。

> [!tip] ClawCodex 项目关联 — Harness Engineering 的完整实现
> **"强制反问机制"** — ClawCodex 的 Clarification 三通道流：
> - **Channel 1（本地）**— operator 在 `.orchestrator_control/` 写文件回答，5s 内 agent 读到
> - **Channel 2（远程）**— agent 在 issue 里 `@mention` 作者提问，等最多 72h
> - **Channel 3（超时升级）**— 等不到回答就走 escalation policy（skip / mark_failed / notify）
> 
> 关键是反问之后的超时策略和优先级。如果本地 operator 和远程 author 同时回答，`simultaneous_grace_ms=5000` 给一个 5 秒窗口合并两个答案，operator 优先。
> 
> **"纯净沙盒"** — `permission_mode`（default / acceptEdits / bypassPermissions）+ `ApprovalPolicy` + `turn_sandbox_policy` 的 `writableRoots` 限制 agent 只能写工作目录。
> 
> **防 LLM 自触发** — F-39 的 `_is_command_author_eligible()` 只允许 issue 作者或 maintainer 发 `/agent retry`，防止 LLM 自己在评论里写 `/agent retry` 触发无限循环。

Q19：（追问）你觉得在应用态，验证环节重要吗？

A：训练态很重要，应用态极难。因为 C 端用户自己都不知道确切想要什么，无法给出标准 Feedback 信号。

> [!tip] ClawCodex 项目关联 — 验证环节的 Proxy Metric 困境
> 我们刚实测出来的例子：agent 正确创建了 `hello.py`，但验证层（noop detection）判定它"什么都没做"然后 force-complete 了。根因是 `get_file_status()` 依赖 git，sequential workspace 没有 `.git`。
> 
> Orchestrator 里有三层验证，每层都有不同的失败模式：
> 1. **test_command** — 跑 `pytest`，但只能验证"没挂"，不能验证"做对了"
> 2. **noop detection** — 用文件变更作为 proxy metric，但对非 git 目录误报
> 3. **review gate** — 最终靠人，但人的带宽是瓶颈
> 
> 面试话术：*"我实测发现，应用态验证的核心难点不是没有 ground truth，而是 proxy metric 的 precision/recall 权衡 — 检测太严会误杀正确结果，检测太松会放过空转。"*

Q20：平时看 AI 方向的论文吗？

A：看。看了关于 Agent Benchmark 长链路任务目标漂移的 Paper，以及各模型的评测技术报告。今年 AI Agent 的重点应该是长程任务和自进化。

> [!tip] ClawCodex 项目关联 — 长链路任务目标漂移的三层防护
> 这就是 Orchestrator 里 AgentRunner 最核心的工程问题。代码里有三层防漂移机制：
> 
> - **Stagnation guard** (`agent_runner.py` `max_no_op_turns=3`) — 连续 N 轮 LLM 没有产出任何 tool call，判定停滞
> - **Loop detection** (`loop_detection_window=5, threshold=3`) — 最近 5 轮内相同 tool-call 签名出现 3 次，判定死循环
> - **Read-only spiral guard** (`_MAX_READ_ONLY_TURNS=4`) — 连续 4 轮只调 Read/Bash/Grep 没有任何 Write/Edit，判定陷入调查死循环
> 
> 刚才实测就撞了：agent 确实创建了 `hello.py`，但 noop detection 还是误杀了它。这就是目标漂移检测的 **false positive 问题** — 论文里讨论的是 agent 偏离目标，工程上要解决的是怎么区分「agent 在思考」和「agent 在空转」。
> 
> 面试话术：*"我在做的项目里就实现了三层漂移防护，但最近测试发现 noop 检测在非 git workspace 下有误报，因为文件变更检测依赖 `git status`，没有 `.git` 的目录恒返回空 —— 这就是 guard 的 precision/recall 权衡。"*
